## Why claude-trace?

Most APM tools treat LLM calls as opaque HTTP requests. You see 200ms of latency to `api.anthropic.com`
— but not *which turn* of a multi-step agent, not *cumulative cost per session*, not *tool calls
nested as children of the turn that invoked them*.

claude-trace is built on the premise that **agent reasoning has structure**. A session has turns.
Turns have tool calls. Tool calls have outcomes. Your observability tooling should reflect that
structure natively as OpenTelemetry spans — not as log lines you grep through.

<CardGrid>
  <Card title="Zero code changes" icon="puzzle">
    One `instrument()` call patches both sync and async Anthropic clients globally. All existing `messages.create()` calls are automatically traced. No decorators, no wrappers, no agent framework needed.
  </Card>
  <Card title="First-class span hierarchy" icon="list-format">
    Every agent decision becomes a structured OTel span. Sessions contain turns, turns contain tool calls. Query in Jaeger, Grafana Tempo, Honeycomb, or any OTLP-compatible backend.
  </Card>
  <Card title="Cost tracking built in" icon="approve-check">
    Token counts and USD cost estimates on every session and turn span. Supports all 16 Claude models including prompt caching (creation at 125%, reads at 10%). Per-customer cost attribution out of the box.
  </Card>
</CardGrid>

## 30-second quickstart

```bash
pip install claude-trace
```

```python
import claude_trace
import anthropic

# One line. Patches both Anthropic() and AsyncAnthropic().
claude_trace.instrument()

client = anthropic.Anthropic()

with claude_trace.session(customer_id="acme", tags=["prod"]) as sess:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": "Write a bash script to list Python files."}],
    )

print(f"Session cost: ${sess.cost.total_usd:.4f}")
print(f"Input tokens: {sess.cost.input_tokens}")
print(f"Output tokens: {sess.cost.output_tokens}")
```

That's it. No configuration required. Spans are exported via whatever `TracerProvider` is registered globally.

## Connecting to your backend

Point your OTel exporter at any compatible backend:

```python
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from claude_trace.config import TraceConfig

provider = TracerProvider()
provider.add_span_processor(
    BatchSpanProcessor(OTLPSpanExporter(endpoint="http://localhost:4317"))
)

claude_trace.instrument(config=TraceConfig(tracer_provider=provider))
```

Works with: **Jaeger**, **Grafana Tempo**, **Honeycomb**, **Datadog**, **AWS X-Ray**, **Google Cloud Trace**, and any OTLP-compatible receiver.
